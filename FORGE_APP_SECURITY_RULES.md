# SG Forge App Security Rules

These rules govern the development, registration, execution, and security review of applications (Forge Apps) integrated into the SG Forge platform.

---

## 1. Iframe Sandboxing & Isolation
* **Sandbox Policy:** All Forge Apps must be rendered inside an `iframe` with the following attributes:
  ```html
  <iframe src="..." sandbox="allow-scripts allow-forms" />
  ```
* **Prohibited Attributes:** 
  * `allow-same-origin` is strictly forbidden. Including it allows the app to bypass the origin boundary and read parent cookies.
  * `allow-top-navigation` is prohibited except for explicit redirect flows approved during registration.
* **Origin Separation:** The micro-app server MUST run on a distinct port (or domain) from the portal core to enforce natural browser origin security blocks.

---

## 2. PostMessage Communication Security
* **Origin Checking:** When sending message payloads (such as theme updates or navigation commands) via the client-side `postMessage` API, applications and the host portal MUST validate the event origin:
  ```javascript
  window.addEventListener('message', (event) => {
    if (event.origin !== expectedOrigin) {
      console.warn("Rejected message from unauthorized origin:", event.origin);
      return;
    }
    // Process message
  });
  ```
* **Structured Payloads:** All client communications must use structured, JSON-serializable payloads containing a descriptive `type` field. Avoid parsing raw strings or executing arbitrary code (e.g. `eval`) from message data.

---

## 3. Token-Level Access Control & Exchanges
* **Handshake Expiration:** Authorization codes (`forge_auth_codes`) generated by the host portal are short-lived. They must be exchanged within **30 seconds** of issue. Expired codes are automatically deleted and rejected.
* **Single-Use Enforcement:** Every authorization code is strictly single-use. If a code is presented more than once, all access tokens associated with that code are invalidated immediately.
* **Token Boundaries:**
  * Access tokens issued to micro-apps are constrained strictly to the scopes approved in the application's `app.json` manifest.
  * A micro-app cannot call portal core APIs outside its declared scope boundaries (e.g., an app without `audit.write` scope will receive a `403 Forbidden` if it attempts to post logs).
  * Access tokens have a maximum lifetime of **60 minutes**, requiring apps to obtain fresh tokens via handshake reload.

---

## 4. Database Schema Isolation Rules
* **Namespace Isolation:** Apps that request database storage (`requiresIsolatedSchema: true`) must execute queries strictly within their assigned schema namespace (e.g. `forge_reference_expenses`).
* **Cross-Schema Queries:** Forge apps are strictly blocked from querying or modifying tables in other namespaces or the core portal schemas (e.g. `users`, `system_logs`). Any database interaction must proceed via standard REST APIs using their assigned access tokens.
